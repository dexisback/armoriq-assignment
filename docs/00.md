# Assignment Summary

The original assignment asks for building a miniature version of ArmorIQ.

ArmorIQ itself is an AI Agent Security Platform.

Rather than building an AI assistant, the objective is to build the layer that sits **between** an AI model and the external tools it can invoke.

Instead of asking:

> Can the model call tools?

the assignment asks:

> Should the model be allowed to call this tool?

That distinction is the heart of the project.

---

# The Core Problem

Modern LLMs can use tools.

For example:

User:

> Restart production server.

The model may decide:

```
restart_server(server="prod-01")
```

Without guardrails, the tool executes immediately.

That introduces problems.

Examples:

* deleting production data
* restarting critical infrastructure
* leaking secrets
* prompt injection
* bypassing system prompts
* calling expensive APIs indefinitely
* escaping intended permissions

The assignment is about preventing that.

---

# Core Philosophy

The project is built around one central idea.

The LLM should **never** be trusted to decide what it is allowed to do.

Instead:

```
User

↓

LLM

↓

Policy Engine

↓

MCP Tools
```

The Policy Engine becomes the single source of truth.

The model may *request* tool execution.

The policy engine decides whether execution is actually allowed.

---

# Project Goal

The goal is not simply to connect to MCP servers.

The goal is to build an autonomous agent that can:

* discover tools dynamically
* reason about which tool to call
* request tool execution
* have every request inspected
* enforce security policies
* optionally require human approval
* execute tools only if permitted
* produce complete audit logs

# Why This Architecture?

Many implementations place security checks inside the agent loop.

Example:

```
if(tool == dangerous){
...
}
```

This architecture was intentionally avoided.

Instead:

```
Agent

↓

Policy Engine

↓

Decision

↓

Tool
```

This makes the policy engine:

* reusable
* independently testable
* transport independent
* LLM independent

The agent does not know how security decisions are made.

The policy engine does not know which LLM is being used.

The MCP registry does not know why a tool was denied.

Every subsystem has one responsibility.

# Complete System Architecture

This chapter explains every major subsystem that exists inside the project and, more importantly, why it exists. While the first chapter focused on the philosophy behind the system, this section focuses on the actual architecture that was implemented.

The goal is that after reading this document, someone should understand how every major component interacts before looking at a single line of implementation.

---

# Monorepo Architecture

The project was intentionally designed as a monorepo.

Instead of placing every piece of code inside a single application, responsibilities are split into reusable packages.

High-level structure:

```text
apps/
    agent/
    custom-mcp/
    dashboard/

packages/
    db/
    logger/
    mcp-registry/
    policy-engine/
    shared-types/
```

Every package has a single responsibility.

This makes the system significantly easier to maintain, extend and test.

---

# Why a Monorepo?

Initially it would have been easier to place everything inside one Express application.

That approach was intentionally avoided.

Reasons:

• Different systems evolve independently.

• Multiple applications consume the same packages.

• Shared types eliminate duplicated interfaces.

• Individual packages remain independently testable.

• Clear ownership boundaries naturally emerge.

The project therefore resembles how a production backend would typically be organized.

---

# Applications

There are three major applications.

---

## 1. Agent

The Agent is the runtime of the entire system.

Responsibilities include:

* receiving user prompts
* communicating with the LLM
* executing the tool loop
* invoking the policy engine
* executing MCP tools
* writing audit logs
* exposing REST APIs
* orchestrating approvals

The agent intentionally does not implement business rules itself.

Instead it delegates those responsibilities to dedicated modules.

Think of the Agent as the conductor of an orchestra.

It coordinates.

It does not perform every task itself.

---

## 2. Dashboard

The Dashboard is an administrative console.

It is not another AI application.

Instead it exists to configure the running agent.

It allows administrators to:

* create rules
* enable or disable rules
* inspect tools
* override risks
* approve requests
* inspect logs
* monitor MCP servers

The dashboard never talks directly to MCP servers.

It communicates only with the Agent API.

---

## 3. Custom MCP Server

The assignment required implementing at least one MCP server.

Instead of choosing a trivial example, the server was designed to resemble a small infrastructure management API.

It exposes several tools including:

* list servers
* restart server
* deploy release
* rollback release
* retrieve logs

The important architectural decision is that this server follows the MCP specification properly.

The Agent does not contain any custom code for these tools.

It discovers them dynamically.

# Prompt Security Layer

Before reaching the LLM, prompts are scanned using heuristic detection.

Current detection covers phrases including:

* ignore previous instructions
* act as root
* bypass security
* override policy
* jailbreak
* disable guardrails

Current behaviour:

Detection

↓

Audit Log

↓

Continue

The system intentionally does not block prompts.

Reason:

False positives are common.

Logging provides visibility without harming legitimate users.

This design mirrors many production security systems where alerts are separated from enforcement.

---

# LLM Layer

The project intentionally supports multiple providers.

Primary provider:

Gemini Flash

Fallback:

Groq Llama 3.3

Routing strategy:

```text
Gemini

↓

Success

↓

Done
```

If Gemini fails:

```text
Gemini

↓

Retry

↓

Retry

↓

Retry

↓

Fallback

↓

Groq
```

Exponential backoff is applied before switching providers.

This significantly improves reliability while maintaining a simple architecture.

# Registry Cache

The registry maintains an in-memory cache.

Purpose:

Avoid rediscovering tools on every request.

Each registry entry contains:

* server configuration
* discovered tools
* connection state
* last synchronization timestamp

Refresh operations simply overwrite the cached tools.

# Tool Catalog

Runtime discovery alone is insufficient.

The project also persists discovered tools.

Reasons:

* dashboard browsing
* historical tracking
* risk overrides
* metadata
* future analytics

Every discovered tool is synchronized into the ToolCatalog table.

This effectively creates an inventory of all available capabilities.

# Rule Cache

Policies are evaluated frequently.

Querying the database for every tool call would be wasteful.

Instead:

Validated rules are cached in memory.

Redis invalidates the cache whenever changes occur.

This provides:

* high performance
* immediate propagation
* simple architecture

```text
↓
```


Invalid rules never reach the policy engine.

This dramatically improves robustness.

# Rule Evaluation

Policies execute in priority order.

Example:

Priority 1

Block Restart

Priority 10

Require Approval

Restart request.

Result:

Block.

Approval never executes.

The first matching rule wins.

This behaviour is deterministic.

---

# Why Priority Exists

Without priority:

Conflicting policies become ambiguous.

Example:

Rule A

Block restart.

Rule B

Require approval.

Which wins?

Priority removes ambiguity.

Lower integer.

Higher precedence.

Simple.

Deterministic.

## Conflicting Rules

Scenario:

# Major Engineering Decisions

Several design decisions significantly influenced the quality of the final architecture.

---

## Policy Engine is the Heart

Many AI applications revolve around the model.

This project intentionally revolves around the policy engine.

The LLM becomes replaceable.

Policies remain authoritative.

---

## Runtime Configuration

Policies belong in the database.

Not TypeScript.

This allows non-developers to govern the system.

---

## Shared Types Everywhere

Every contract originates from shared-types.

Benefits:

Single source of truth.

Compile-time consistency.

Runtime validation.

---

## Zod Validation Boundary

One of the most important architectural improvements.

Database JSON

↓

Validation

↓

Engine Types

↓

Execution

Invalid configurations never enter runtime.

---

## Service-Oriented Backend

Instead of enormous controllers:

Rule Loader.

Risk Resolver.

Tool Loop.

Chat Service.

Registry.

Each owns one responsibility.

---

## Infrastructure Agnostic Packages

Packages avoid framework knowledge.

Policy Engine.

Registry.

Shared Types.

Could all be reused in another application.

# Final Philosophy

This project ultimately became much more than "an AI agent."

It became a governance layer for autonomous systems.

The most important architectural shift during development was changing the perspective from:

"How can an LLM use tools?"

to:

"Who decides whether the LLM is allowed to use tools?"

Everything in this repository ultimately supports that single principle.

The LLM proposes.

ArmorIQ decides.

Engineering Handbook

This final section serves as the engineering handbook for the project.

Rather than explaining implementation details, it explains how to think about the system, why it was designed this way, and how it should evolve in the future.

Mental Model

Every engineer working on this repository should keep one mental model in mind.

The project is NOT:

an AI chatbot
an MCP client
a dashboard

The project IS:

A governance layer that controls autonomous AI systems.

Everything else exists to support that idea.

The LLM is not the center of the architecture.

The Policy Engine is.

Project Principles

Throughout development several principles remained constant.

1. Security Before Capability

The goal is not enabling tools.

The goal is enabling tools safely.

Whenever capability and safety conflict, safety wins.

2. Runtime Configuration

Nothing security-related should require changing source code.

Administrators should govern the system using:

policies
approvals
risk overrides
dashboard controls

The code should remain unchanged.

3. Configuration Is Data

Policies are data.

Risk overrides are data.

Tool metadata is data.

Approvals are data.

Nothing should be hardcoded inside TypeScript.

4. Everything Has One Owner

Every subsystem owns exactly one concern.

Examples:

Policy Engine

Owns authorization.

Registry

Owns discovery.

Prompt Security

Owns prompt inspection.

Rule Loader

Owns loading rules.

Approval Service

Owns approvals.

Tool Loop

Owns orchestration.

This dramatically reduces coupling.

System Ownership

Every major folder has a single responsibility.

apps/

Application entrypoints.

packages/

Reusable libraries.

generated/

Generated code only.

Never place business logic inside generated code.

Backend Ownership

Agent owns:

Conversation.

Policy Engine owns:

Decisions.

Registry owns:

Tools.

Database owns:

Persistence.

Dashboard owns:

Administration.

Redis owns:

Synchronization.

This separation should never be violated.

Data Flow Philosophy

Data always moves downward.

Example:

Why There Is No Hardcoding

Hardcoding policies creates two problems.

First:

Security changes require deployment.

Second:

Non-developers cannot administer the system.

Moving policies into the database solves both.

Why Redis Exists

Redis was not introduced for caching.

Redis exists solely to synchronize runtime state.

Without Redis:

Policy changed.

↓

Restart agent.

With Redis:

Policy changed.

↓

Publish.

↓

Reload.

↓

Done.

Why Shared Types Exist

Every package depends on contracts.

Not implementations.

Shared-types ensures:

Compile-time consistency.

Runtime validation.

Documentation.

Discoverability.

Without shared-types, interfaces drift over time.

Why Zod Exists

TypeScript validates only during compilation.

The database stores runtime JSON.

Runtime JSON cannot be trusted.

Therefore:

Database

↓

Zod

↓

Engine

Every runtime boundary validates input.

Why Policy Engine Is Pure

A pure engine is easier to:

Why Tool Catalog Exists

Discovery is transient.

Inventory is permanent.

Persisting discovered tools enables:

Why Audit Logs Exist

Audit logs are not debugging logs.

They answer:

What happened?

Why?

Who decided?

Which policy matched?

Could the action be reproduced?

Production AI systems require explainability.

Why Event Types Exist

A decision and an event are different concepts.

Example:

Decision:

ALLOW

Event:

PROMPT_INJECTION

Separating these makes querying and visualization much easier.

Why Approvals Exist

Some actions should not be blocked.

Some actions should not be automatic.

Approvals introduce a third option.

ALLOW.

DENY.

PAUSE.

This is closer to enterprise security than binary allow/deny systems.

Why Prompt Injection Is Logged

Prompt injection detection intentionally does not block execution.

Reasons:

False positives.

User education.

Security monitoring.

Future policy versions may allow administrators to choose:

Monitor.

Warn.

Block.

Why Risk Is Separate From Policy

Risk describes a tool.

Policy describes organizational behaviour.

Example:

restart_server

Risk:

CRITICAL

One company:

Always block.

Another:

Require approval.

Another:

Allow.

Separating these concepts makes the system much more flexible.

Interview Talking Points

If asked:

"Why did you separate the Policy Engine?"

Answer:

Because authorization should not depend on the transport layer or the LLM provider. A pure policy engine is easier to test, reuse, and reason about.

If asked:

"Why Redis?"

Answer:

Redis is not being used as a cache. It is used as an event bus to propagate policy changes to running agents without requiring a restart.

If asked:

"Why dynamic discovery?"

Answer:

Hardcoding tools violates the MCP philosophy. The registry should discover whatever capabilities a compliant MCP server exposes.

If asked:

"Why persist discovered tools?"

Answer:

Runtime discovery is ephemeral. Persisting tools enables inventory management, historical tracking, risk overrides, and dashboard visualization.

If asked:

"Why Zod if TypeScript already exists?"

Answer:

TypeScript validates source code. It cannot validate runtime JSON loaded from the database. Zod protects the runtime boundary.

If asked:

"What is the biggest architectural decision?"

Answer:

Treating the Policy Engine—not the LLM—as the center of the system.

Everything else follows from that decision.

Final Statement

The finished project is intentionally larger than the original assignment.

Rather than building a chatbot with tools, the objective became building a modular governance platform for autonomous AI systems.

Every major architectural decision—from package boundaries to Redis synchronization, runtime validation, dynamic discovery, audit logging, and approvals—supports that vision.

The guiding principle throughout the project can be summarized in one sentence:

The LLM proposes an action. ArmorIQ decides whether that action is permitted.

Everything else in the repository exists to make that decision observable, configurable, explainable, and enforceable.
